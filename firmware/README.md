# Firmware — ESP32 Sensor Node

Firmware for the physical bin unit: an ESP32 DevKit V1 reading an AJ‑SR04M ultrasonic sensor and publishing telemetry to the platform over MQTTS.

## Hardware

- ESP32 DevKit V1 (30-pin)
- AJ‑SR04M ultrasonic distance sensor, used in **Trigger/Echo mode** (the same 2-pin mode as a classic HC‑SR04)
- Continuous 5V power (Micro‑USB or a 5V supply) — this firmware does not use deep sleep and expects to run permanently powered

### Wiring

| AJ‑SR04M pin | ESP32 DevKit V1 pin |
|---|---|
| VCC | 5V (VIN) |
| GND | GND |
| TRIG | GPIO12 |
| ECHO | GPIO13 |

Pin numbers are defined in `include/config.h` (`TRIGGER_PIN`, `ECHO_PIN`) if you need to wire it differently.

## How It Works

- `src/main.cpp` runs a non-blocking loop: it keeps Wi‑Fi and the MQTT connection alive, and on a timer (or on demand — see Remote Commands below) triggers a measurement cycle.
- Each measurement cycle takes a burst of 11 raw ultrasonic pulses, discards physically implausible readings, and reduces the burst to a single robust distance using a median + tolerance filter. This is the "Stage 1" noise filter; the stateful part of the algorithm (rejecting transient false readings across cycles, and the fill-can-only-increase-until-emptied ratchet) runs server-side in `fill-estimation-service`.
- The filtered distance is published as JSON to `wastebin/{ZONE_ID}/{DEVICE_ID}/telemetry`.
- On connect, the device publishes `online` (retained) to `wastebin/{ZONE_ID}/{DEVICE_ID}/status`, with `offline` configured as the MQTT Last Will so the backend can detect a dead/disconnected device automatically.
- The device subscribes to `wastebin/{ZONE_ID}/{DEVICE_ID}/cmd` for remote commands (see below).

### MQTT Topics

| Topic | Direction | Purpose |
|---|---|---|
| `wastebin/{ZONE_ID}/{DEVICE_ID}/telemetry` | publish | Distance reading + metadata, sent every `measureIntervalMs` |
| `wastebin/{ZONE_ID}/{DEVICE_ID}/status` | publish (retained + LWT) | `online` / `offline` |
| `wastebin/{ZONE_ID}/{DEVICE_ID}/cmd` | subscribe | Remote commands, see below |

### Remote Commands

Publish a JSON payload to the device's `cmd` topic:

```json
{ "command": "force-report" }
```
Triggers an immediate measurement cycle instead of waiting for the timer.

```json
{ "command": "set-interval", "value": 15000 }
```
Changes the measurement interval at runtime (`value` in milliseconds, clamped between 1,000 and 3,600,000).

## Building and Flashing

This project uses [PlatformIO](https://platformio.org/). With the PlatformIO CLI or the VS Code extension installed:

```bash
cd firmware
pio run                 # build
pio run --target upload # flash over USB
pio device monitor      # view serial logs (115200 baud)
```

All library dependencies (`PubSubClient`, `ArduinoJson`) are declared in `platformio.ini` and are fetched automatically on first build.

## Configuration (`secrets.cpp`)

Device-specific and network credentials are kept out of version control. Before building:

```bash
cp src/secrets.cpp.example src/secrets.cpp
```

Then edit `src/secrets.cpp` and fill in every field:

| Field | Value |
|---|---|
| `DEVICE_ID` | Unique ID for this bin, e.g. `bin-0142`. **Must exactly match** the `bin_id` you register through the dashboard/API — telemetry from an unregistered `device_id` is silently dropped by the backend. |
| `ZONE_ID` | The zone/district this bin belongs to, e.g. `district-7`. Must match the `zone_id` used when registering the bin. |
| `WIFI_SSID` / `WIFI_PASSWORD` | Credentials for the local Wi‑Fi network the bin will connect through. |
| `MQTT_SERVER` | Hostname or IP of the broker, reachable from wherever the device is physically deployed. |
| `MQTT_PORT` | `8883` for the TLS listener (recommended/default). |
| `MQTT_USER` / `MQTT_PASSWORD` | The `bin_device` account credentials — `MQTT_PASSWORD` must match `MQTT_BIN_PASSWORD` from the backend's `.env` (this is what `setup_security.py` hashes into the broker's password file). |
| `MQTT_CA_CERT` | The contents of `mosquitto/certs/ca.crt`, generated by running `setup_security.py` in the project root. Open that file and paste its full contents (including the `BEGIN`/`END CERTIFICATE` lines) between the `R"EOF( ... )EOF"` markers, preserving line breaks. |

`src/secrets.cpp` is listed in `.gitignore` — only `src/secrets.cpp.example` (with placeholder values) should ever be committed.

## Security Notes

- Never commit `src/secrets.cpp`. If real Wi‑Fi or MQTT credentials are ever accidentally committed, shared, or pasted somewhere outside your control, treat them as compromised and rotate them (change the Wi‑Fi password, regenerate the MQTT password via `setup_security.py`) rather than just deleting the file.
- The firmware calls `setInsecure()` on the TLS client, so the MQTTS connection is encrypted but does **not** verify the broker's certificate against `MQTT_CA_CERT`. This is fine on a trusted local/private network; if the broker is ever reachable from an untrusted network, switch to proper certificate validation before relying on it.

## Troubleshooting

- **Stuck at "Connecting to SSID..."**: double-check `WIFI_SSID`/`WIFI_PASSWORD`, and confirm the network is 2.4 GHz (the ESP32 does not support 5 GHz Wi‑Fi).
- **MQTT connection fails repeatedly (`rc=` printed on serial)**: verify `MQTT_SERVER`/`MQTT_PORT` are reachable from the device's network, and that `MQTT_USER`/`MQTT_PASSWORD` match an account created by `setup_security.py`.
- **Sensor read fault printed every cycle**: check the TRIG/ECHO wiring and that the sensor has a clear line of sight — readings outside `[SENSOR_BLIND_ZONE_CM, BIN_DEPTH_CM + 10]` (see `include/config.h`) are discarded, and a majority-invalid burst is reported as a fault rather than a guess.
